# 이슈 작업 자동화 가이드

> 다음 이슈를 Codex와 함께 진행할 때 재사용하는 작업 규칙 문서
> 최근 작업 패턴은 `#43 자리배치`, `#44 시간표` 이슈를 기준으로 정리한다.

---

## 목적

이 문서는 다음 목적을 가진다.

- 이슈를 받을 때 어떤 기준 브랜치에서 시작할지 빠르게 판단한다.
- 템플릿, 운영 페이지, 기존 PR 패턴을 먼저 참고하도록 강제한다.
- 프론트/백엔드 변경을 억지로 섞지 않고 실제 변경 범위대로 커밋을 나눈다.
- PR 설명을 최근 저장소 스타일과 동일하게 맞춘다.

---

## 시작 전 필수 확인

프론트 작업이 포함되면 아래 문서를 먼저 읽는다.

- `AGENTS.md`
- `.impeccable.md`
- `front-end/REFACTOR_GUIDE.md`
- `docs/개발가이드/BRANCH_REFACT_GUIDE.md`

추가로 반드시 확인한다.

- 현재 이슈 번호
- 기존 관련 PR이 이미 열려 있는지
- 실제 운영 페이지가 있는지
- 비교용 계정이 있는지
- master 또는 기존 Django 템플릿 기준 구현이 남아 있는지

---

## 브랜치 시작 규칙

기본 원칙:

- 독립 이슈면 `refact_front_to_nextjs` 기준으로 새 브랜치 생성
- 이전 이슈 변경을 그대로 이어받아야 하면 직전 작업 브랜치에서 분기
- stacked PR이 필요한 경우 base 브랜치는 직전 이슈 브랜치로 설정

예시:

- `#43 자리배치`는 `refact_front_to_nextjs` 기준
- `#44 시간표`는 `feat/#43-seats` 기준으로 분기 후 stacked PR 생성

판단 기준:

- 시간표가 자리배치 모달 데이터를 그대로 재사용하면 `#43` 위에서 분기
- 기능이 완전히 독립이면 `refact_front_to_nextjs`에서 새로 분기

---

## 구현 전 조사 순서

다음 순서로 먼저 읽고 시작한다.

1. 이슈 본문 확인
2. 최근 유사 PR 확인
3. Django 템플릿 또는 master 구조 확인
4. 운영 페이지 직접 확인
5. 로컬 현재 구현 상태 확인

반드시 지킬 점:

- 운영 페이지가 있으면 실제 화면과 데이터 구조를 먼저 확인한다.
- 템플릿에 있는 데이터가 Next 쪽에서 빠지면 안 된다.
- 구조만 비슷하게 만드는 것이 아니라 입출력 데이터 흐름도 맞춰야 한다.
- 실제 운영 구조와 다르면 템플릿보다 운영 동작을 우선 참고하되, 왜 다른지 확인한다.

예시:

- `quiz`는 템플릿의 입력/출력 구조를 유지
- `seats`, `timetable`은 운영 `https://cartellab-production.up.railway.app/seats/`를 직접 비교
- 비교 계정 `test / test2323`로 로그인 후 반별 시간표 구조 확인

## 프론트 구현 규칙

`front-end/REFACTOR_GUIDE.md`를 기준으로 다음 구조를 따른다.

```text
front-end/app/[page]/
├── page.tsx
├── _hero-section.tsx
├── _[name]-section.tsx
└── _styles.ts
```

규칙:

- `page.tsx`는 fetch, 인증, 섹션 조립만 담당
- 섹션 내부 상태는 섹션 파일에 둔다
- 페이지 전용 `_types.ts`는 만들지 않는다
- 반복 데이터는 `constants/`로 분리한다
- 공용이 아닌 UI는 해당 섹션 파일 내부에서 해결한다

프론트 디자인 규칙:

- `.impeccable.md` 우선
- Pretendard 기반
- 넓은 여백, 강한 타이포 계층
- 과도한 카드 중첩 금지
- 오렌지는 CTA/강조 시점에만 사용

## 백엔드 구현 규칙

`refact_front_to_nextjs` 계열 작업에서는 Django는 API 역할이 우선이다.

규칙:

- 새 기능이 필요하면 가능하면 `/api/...` 라우트로 분리
- HTML 템플릿 뷰는 유지하고, Next에서 쓸 JSON API를 별도로 추가하는 방향을 우선 검토
- 직접 화면 렌더링용 구조를 Next 쪽으로 옮길 때는 응답 데이터 구조를 먼저 고정

주의:

- 실제로 백엔드 변경이 최종적으로 필요 없으면 커밋을 억지로 만들지 않는다.
- 탐색 중 임시로 만진 백엔드 변경은 최종 방향이 아니면 되돌린다.

예시:

- `#44 시간표`는 초기에는 API를 검토했지만, 최종적으로 운영 `seats` 모달 구조를 프론트 상수 기반으로 맞추는 방식으로 정리했고 백엔드 커밋은 만들지 않았다.

## 커밋 분리 규칙

기본 원칙:

- `chore` / `backend` / `frontend` 순으로 나눈다
- 단, 실제 변경이 없는 레이어는 커밋하지 않는다
- 프론트 안에서도 역할이 다르면 추가로 분리한다

권장 패턴:

1. `chore : ...`
2. `feat : ... API 구현`
3. `feat : ... 페이지 구현`
4. `refact : 기존 페이지 연동 구조 개선`

실전 예시:

### #43 자리배치

- `chore : 로컬 문서 및 codex 파일 gitignore 추가`
- `feat : 좌석 API 구현`
- `feat : 자리배치 페이지 구현`

### #44 시간표

- `feat : 시간표 라우트 및 데이터 상수 추가`
- `feat : 시간표 페이지 구현`
- `refact : 자리배치 시간표 모달 구조 개선`

하지 말 것:

- 변경이 없는데 프론트/백엔드로 억지 분리
- unrelated 파일을 섞어서 한 커밋으로 몰아넣기
- 사용자 확인 전에 임의로 커밋/푸시 강행

---

## 커밋/푸시 타이밍

기본 흐름:

1. 먼저 구현
2. 실제 동작 확인
3. 사용자에게 구조/차이점 설명
4. 커밋 여부 확인
5. 승인 후 커밋 및 푸시

특히 아래 요청이 있으면 반드시 지킨다.

- "커밋하지마"
- "에러부터 보자"
- "파일은 유지하고 커밋만 취소"

즉, 작업 완료와 커밋은 별개 단계로 본다.

## PR 작성 규칙

최근 저장소 패턴을 그대로 따른다.

형식:

```md
closes #이슈번호

## 개요
...

## 백엔드 변경사항
...

## 프론트엔드 변경사항
...

## 기타
...

## 테스트
- [x] ...
- [ ] ...
```

규칙:

- 제목은 최근 저장소 스타일 유지
- 본문 첫 줄에 `closes #번호`
- 테스트는 실제로 한 것만 체크
- 못 한 테스트는 이유와 함께 미체크로 남김
- stacked PR이면 `## 기타`에 명시

예시:

- `#43` PR은 `refact_front_to_nextjs` base
- `#44` PR은 `feat/#43-seats` base로 stacked PR 생성

---

## 운영 페이지 비교 규칙

운영 URL이 있으면 반드시 비교한다.

체크 항목:

- 비로그인/로그인 동작 차이
- 유저 권한별 노출 차이
- 반, 학년, 작성자 등 사용자 속성에 따른 분기
- 텍스트, 버튼, 모달 흐름
- 누락된 데이터가 없는지
- 연속 데이터 병합이나 표시 규칙이 있는지

예시:

- `seats` 시간표는 단순 목록이 아니라 `교시 x 요일` 테이블
- 같은 수업이 연속 교시일 때 하나의 칸처럼 병합되어 보여야 함
- 사용자 `class_group`에 따라 A반/B반 시간표가 달라져야 함

## 작업 중 사용자와 합의할 내용

중간에 아래 사항은 꼭 사용자와 맞춘다.

- 브랜치를 새로 팔지, 현재 브랜치에서 이어갈지
- 커밋을 지금 할지, 나중에 할지
- 프론트/백엔드 분리를 실제 변경 범위대로 할지
- 운영 페이지 기준으로 맞출지, 템플릿 기준으로 맞출지

단, 합리적으로 판단 가능한 경우에는 멈추지 말고 진행한다.
