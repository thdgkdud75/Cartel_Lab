# 팀원 전달용 ERD 설명서 (한글)

이 문서는 `시각화된 ERD 이미지`와 함께 전달하는 설명서입니다.  
목표는 신입 개발자가 코드를 보지 않아도 DB 구조와 구현 범위를 이해하게 하는 것입니다.

## 1. 프로젝트에서 DB가 하는 일
- 사용자 식별(학번, 이름, 마지막 로그인)
- 사용자별 출결 기록 관리
- 사용자별 주간 계획/개인 목표 관리
- 사용자 좌석 배정 상태 관리

## 2. 엔티티(테이블) 설명

### 2.1 사용자(`users_user`)
- 한 명의 사용자를 나타내는 기준 테이블
- 핵심 정보: 학번, 이름, 역할, 마지막 로그인 시간
- 다른 모든 기능 테이블이 이 테이블을 참조

### 2.2 출결(`attendance_record`)
- 날짜별 출결 상태를 저장
- 한 사용자에게 여러 출결 기록이 생김
- 같은 사용자/같은 날짜는 1건만 허용

### 2.3 주간 계획(`planner_weekly_plan`)
- 사용자별 주 단위 계획의 상위 묶음
- 예: "3월 2주차 계획"

### 2.4 주간 계획 항목(`planner_weekly_plan_item`)
- 주간 계획에 들어가는 할 일 목록
- 완료 여부, 우선순위, 마감일 관리

### 2.5 개인 목표(`planner_personal_goal`)
- 사용자별 장기/단기 목표 관리
- 상태: 진행중(active), 완료(done), 보류(hold)

### 2.6 좌석(`seats_seat`)
- 좌석 코드(A-01 등)와 좌표/활성 여부 저장

### 2.7 좌석 배정(`seats_assignment`)
- 어떤 사용자가 어떤 좌석을 언제부터 배정받았는지 저장
- 현재 배정 여부(`is_current`)로 최신 상태 판단

## 3. 관계(ERD 핵심)
- 사용자 1 : N 출결
- 사용자 1 : N 주간 계획
- 주간 계획 1 : N 주간 계획 항목
- 사용자 1 : N 개인 목표
- 좌석 1 : N 좌석 배정 이력
- 사용자 1 : N 좌석 배정 이력

## 4. 반드시 지켜야 할 규칙
- 출결은 `사용자 + 날짜` 중복 금지
- 학번은 사용자 기준에서 유일해야 함
- 좌석 현재 배정은 동시에 여러 건이 되지 않게 처리
- 모든 기능 테이블은 사용자 FK 무결성 유지

## 5. 페이지 기능과 DB 연결
- 로그인/회원가입 페이지: `users_user` 사용
- 출결 페이지: `attendance_record` 생성/조회
- 주간 계획 페이지: `planner_weekly_plan`, `planner_weekly_plan_item` 사용
- 개인 목표 페이지: `planner_personal_goal` 사용
- 좌석 페이지: `seats_seat`, `seats_assignment` 사용

## 6. 신입 3명 역할 분배 예시
- 신입1: 사용자/인증(회원가입, 로그인, 사용자 조회)
- 신입2: 출결(등록, 수정, 일자별 조회)
- 신입3: 계획+좌석(주간 계획, 목표, 좌석 배정)

## 7. 전달 시 같이 말해줄 가이드 문구
아래 문구를 그대로 전달해도 됩니다.

1. "ERD는 v1 기준이므로 코어 구조(users, 출결 관계)는 유지합니다."
2. "컬럼 추가나 보조 인덱스는 PR로 제안 가능합니다."
3. "컬럼 삭제/타입 변경/FK 변경은 팀 리뷰 후에만 진행합니다."
4. "기능 개발은 페이지 단위로 하되, 사용자 FK 규칙은 반드시 지켜주세요."

## 8. 체크리스트
- [ ] ERD 이미지에 테이블명/관계선/카디널리티가 표시되어 있다.
- [ ] 사용자 기준 참조 관계가 누락되지 않았다.
- [ ] 중복 금지 규칙(학번, 사용자-날짜 출결)이 반영되어 있다.
- [ ] 페이지별로 어떤 테이블을 읽고/쓰는지 팀원들이 이해했다.
